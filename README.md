# go-stemmers

A Go port of the [`rust-stemmers`](https://github.com/CurrySoftware/rust-stemmers)
crate: the [Snowball](https://snowballstem.org/) stemming algorithms with a small
`Algorithm`/`Stemmer` API. Every algorithm produces output **byte-identical** to
rust-stemmers 1.2.0 (and thus to the reference Snowball implementation), verified
word-for-word against rust-stemmers' own output over a CC0 corpus and by a live
differential test against rust-stemmers itself.

All 18 algorithms are ported: Arabic, Danish, Dutch, English, Finnish, French,
German, Greek, Hungarian, Italian, Norwegian, Portuguese, Romanian, Russian,
Spanish, Swedish, Tamil, Turkish.

## Install

```sh
go get github.com/freeeve/go-stemmers
```

Requires Go 1.25+.

## Usage

```go
import "github.com/freeeve/go-stemmers"

s := stemmers.New(stemmers.English)
s.Stem("fruitlessly") // "fruitless"
s.Stem("national")    // "nation"
s.Stem("sky")         // "sky"  (unchanged, returned as-is)
```

- **Input must already be lowercased** — that is the Snowball/rust-stemmers
  contract. (rust-stemmers takes `&str` which is always valid UTF-8; this library
  accepts any Go string and returns invalid UTF-8 unchanged rather than panicking,
  but stemming is only meaningful on lowercased, valid UTF-8 words.)
- `New` panics on an unknown `Algorithm`.
- A `Stemmer` is **safe for concurrent use**: each `Stem` call takes its own
  working state from an internal pool, so one `Stemmer` can be shared across
  goroutines.
- The `Algorithm` enum mirrors rust-stemmers' order one-to-one.

## Why byte-identical output matters

This library exists so a Go service can stem text exactly as a Rust/wasm peer
does. In an inverted index, the indexed term and the query term must reduce to the
*same* stem byte-for-byte, or every stemmed lookup silently misses. go-stemmers is
the build-side mirror of rust-stemmers used by
[roaringrange](https://github.com/freeeve/roaringrange)'s term index: build the
index from Go, query it from the wasm reader, and the stemming agrees.

This is **output-string parity**, not a serialized byte format — a stemmer has no
on-disk representation. The contract is simply that `Stem(w)` returns the identical
string rust-stemmers returns, for every `w`.

## Correctness

The port is a faithful, mechanical translation of rust-stemmers' generated code
(itself generated by the Snowball compiler): each Snowball labelled block
(`'lab: loop { … break 'lab }`) becomes a Go labelled `for { … break LABEL }`, and
the result-code dispatch (`else if among_var == N`) becomes a `switch`. It is held
to byte-for-byte parity by three layers of tests:

- **`TestVectors`** — stems every word of `testdata/corpus.txt` (a CC0
  OpenAlex-derived word list) with each algorithm and asserts it equals
  `res_<lang>.txt`. Those goldens are rust-stemmers 1.2.0 output, generated by
  `rustgen` (see [`testdata/NOTICE.md`](testdata/NOTICE.md)). This runs in CI with
  no Rust toolchain and is the parity teeth: a single divergence fails the build.
- **`TestRustgenParity`** — a cross-language differential check against the *live*
  rust-stemmers oracle (`rustgen`) over the corpus, for all 18 algorithms. It runs
  only when `rustgen` is built (CI's oracle job builds it), so the pure-Go test
  stays hermetic.
- **`FuzzStem`** — robustness: no algorithm may panic on arbitrary input, and a
  stem of valid-UTF-8 input stays valid UTF-8. (Length is *not* invariant — the
  Turkish algorithm can append a vowel — so byte-for-byte output is asserted by the
  vectors and the differential test, not here.)

```sh
go test ./...                 # vectors + known cases + fuzz seed corpus
go test -race ./...           # also exercises the concurrency-safe pool
```

## Performance

Stemming is allocation-light and within ~15% of native Rust.

- **Allocations:** a `Stemmer` pools its working buffer and per-run state, and
  returns the input string unchanged when a word does not stem. So an unchanged
  token costs **0 allocations**, and a stemmed token costs exactly **one** (the
  result string — unavoidable, since the pooled buffer is reused). English over a
  mixed corpus measures ~0 allocs/op, 4 B/op.
- **Throughput vs Rust:** on English over the same 60k-term OpenAlex corpus
  (Apple M3 Max, interleaved runs to cancel thermal drift), go-stemmers runs at
  **~85–90% of rust-stemmers' throughput** — about 1.15× the time per word
  (~256 ns/word vs ~220 ns/word). The remaining gap is the usual Go-vs-Rust tax:
  bounds checks in the hot `FindAmongB` suffix-table search, the generic-dictionary
  indirection on `FindAmong[T]`, and GC of the one result string per stemmed word.

```sh
go test -bench=English -benchmem        # OpenAlex corpus if present, else the vocab
go test -bench=All                      # per-language throughput
```

### Reproducing the Go-vs-Rust comparison

Build the Rust probe (links rust-stemmers 1.2.0) and run both over the same corpus
(`testdata/corpus.txt` is committed; stream a larger `testdata/bench.corpus` for a
beefier run — both the benchmark and the differential test prefer it when present):

```sh
cargo build --release --manifest-path rustgen/Cargo.toml

# Rust reference
./rustgen/target/release/bench english testdata/corpus.txt 300

# Go
go test -bench=English -benchtime=2s
```

## How it's structured

- `internal/snowball` — the runtime (`Env`, `Among`, `FindAmong`/`FindAmongB`): a
  byte-faithful port of rust-stemmers' `snowball_env.rs`/`among.rs`, shared by
  every algorithm.
- `internal/<lang>` — one package per language, each a faithful translation of the
  generated `algorithms/<lang>.rs`.
- `stemmers.go` — the public `Algorithm`/`Stemmer` API.
- `rustgen/` — a small Rust crate linking rust-stemmers 1.2.0: the **oracle** that
  generates goldens, backs the live differential test, and provides the `bench`
  throughput probe.
- `cmd/oacorpus/` — streams the public OpenAlex Works snapshot (CC0) into a word
  corpus for benchmarking and the differential test; nothing is vendored.

## Running the differential test

```sh
cargo build --release --manifest-path rustgen/Cargo.toml
go test -run TestRustgenParity          # over the committed corpus
# optional, for depth:
go run ./cmd/oacorpus -budget 200000 -out testdata/bench.corpus
go test -run TestRustgenParity
```

## Pinned version & regenerating goldens

The parity contract is pinned to **rust-stemmers 1.2.0** (`rustgen/Cargo.toml`).
The committed test data is a CC0 OpenAlex-derived input corpus (`testdata/corpus.txt`)
plus per-language goldens generated by `rustgen` over it — no third-party
vocabularies are vendored. Bumping the pinned version means regenerating the
goldens; see [`testdata/NOTICE.md`](testdata/NOTICE.md) for the exact commands and
provenance.
